    [x] 11.9.4 Strengthen the merge for live monitoring. Today
        mk_history_record dedups against only the instance's single
        newest entry, and entries are ordered by arrival (prepend), not
        by observation time. A future live-monitoring feature means
        several mcp-kodi processes observing the same playback and
        recording it at slightly different times; their findings must
        merge into one event even when observations arrive late or out
        of order (process B's older sighting of X landing after process
        A already logged X then Y must not spawn a spurious second X).
        Match the incoming observation against a recent per-instance
        time window instead of the single newest entry, insert by the
        observation's `at` timestamp, and on a match merge rather than
        skip (keep the earliest `at` plus a last_seen field) — so
        recording stays idempotent under concurrent flock-serialized
        writers. Landed: the instance's newest entry still matches on
        item identity alone (same continuous playback hours later stays
        one play, the call-driven semantics), older entries merge only
        within a 10-minute window of their [at, last_seen] coverage;
        the observation stamp is taken before the flock; a merge that
        changes nothing skips the rewrite.
    [x] 11.9.5 Live monitoring — poll the Kodi instances so the history
        also captures playback not initiated through the MCP server
        (e.g. started from the TV remote). Poll every ~2 minutes by
        timer (websocket notifications may or may not be available, so
        don't depend on them): query each configured instance's active
        player + current item and feed it through mk_history_record,
        which keeps updating the history and dedups repeat sightings.
        An unreachable box is normal (powered off) and must be skipped
        silently. This is the feature 11.9.4's merge hardening prepares
        for — with several mcp-kodi processes polling the same boxes,
        the merge must stay idempotent. Landed:
        mk_tools_poll_history() snapshots every configured instance via
        player_state() — the same path every tool call takes, history
        feed included — dropping errors without a warning; an idle box
        costs one RPC and never touches the log. main() arms the
        recurring timer on the stdio main loop: 120s default,
        MCP_KODI_POLL_SECONDS overrides (0 disables; tests use 1s).
        Coverage is deliberately session-lifetime — the log records
        what played while a server was up, not a daemon's audit. Unit
        cases (test-tools-handlers.c) + tests/monitor.test, which
        starts playback behind the server's back over raw JSON-RPC and
        asserts the poller logged it once (merged across rounds) while
        skipping a dead box silently.

    [x] 11.9.5.1 Coarsen the per-poll last_seen rewrite. Each poll of a
        continuously-playing item merged into its newest entry and bumped
        last_seen, rewriting the whole ~22KB log every ~2 minutes (×N for
        N concurrent client-session processes) just to nudge a timestamp —
        a long single play (a movie) could rewrite the file dozens of
        times saying nothing new. Fix (mk-history.c): the newest-entry
        merge now persists a bare last_seen advance only once it clears
        MK_HISTORY_LAST_SEEN_STEP (10 min) past what is stored, so the log
        updates in steps, not once per poll; last_seen trails the true
        latest sighting by at most that step, and a hard exit loses no more
        than that. find_merge_match reports whether the match was the
        instance's newest so only that case is coarsened — a rare
        out-of-order sighting folding into an OLDER entry still records
        last_seen precisely (it is not an amplification source, and its
        freshness aids cross-process convergence). The file stays the
        single source of truth (no in-memory deferral), keeping the
        multi-process merge idempotent. monitor.test now asserts last_seen
        stays deferred across the short poll window; the unit cases use
        minute-scale gaps that clear the step and are unaffected.

[ ] 11.11 Config hot-reload (mk-config). The server loads config.json once
    at startup (main.c) and the one MkConfig is shared by mk-kodi and
    mk-tools, so a hand edit to the file — notably an allow_rpc grant,
    which is hand-edit-only by design — is invisible until restart.
    Detect changes at runtime and reload: either a GFileMonitor on the
    file, or a cheap mtime stat before each tools/call. Must ignore the
    server's own atomic saves from the instances tool (temp + rename
    fires the monitor / bumps the mtime), and must update the existing
    MkConfig in place — mk-kodi and mk-tools borrow the pointer, so
    swapping in a new object would leave them stale (mk-kodi already
    resolves instances per-request, so an in-place update is enough). A
    malformed file on reload should keep the last good config, not kill
    the server.

[ ] 11.13 History tool — optional filters for a more flexible query. Today
    `history` takes only instance/since/until/limit (mk-tools.c
    schema_history / handler_history; the match itself is mk_history_read
    in mk-history.c). It reads only the local history.json and makes no
    Kodi call, so every filter is an app-side predicate over the stored
    entries — cheap to add and composable. Plan to add, all OPTIONAL and
    AND-combined with each other and the existing window:
      - media/kind: restrict by the entry's `media`
        (song/episode/movie/musicvideo/picture/channel) and/or `kind`
        (audio/video) — "what music did I play", "movies only".
      - artist/band: a substring (case-insensitive) over the performer —
        a primary, high-value filter ("everything Pink Floyd I played",
        "how much Metallica this week"). It is the music analogue of the
        searchmedia `artist` / contributors band lookup, fed by the same
        names. UNBLOCKED by 11.13.1 (done): the capture now records an
        album/display-artist fallback, so the entry carries an artist whenever
        Kodi knows one — the filter is only as good as what monitoring captured,
        but that is now as good as Kodi's own metadata.
      - text match: a substring (case-insensitive) over the other human
        fields an entry carries — `title`/`album`/`showtitle`/`label` —
        so a partial title/album narrows the log without the caller
        knowing ids. Decide one combined `match`/`q` across all the text
        fields (artist included) vs. per-field args (artist=/album=/
        title=); lean toward a single substring for simplicity, mirroring
        how searchmedia/contributors already take a loose name — but keep
        a dedicated artist arg if music-by-performer is the common query.
      - id: exact library-id match for "every time THIS item played",
        paired with media (ids are per-type), feeding back a searchmedia
        or getplaylist row's id.
      - count: count-only mode (total, zero entries) like searchmedia's
        `count`, for "how many times / how much did X play" without
        hauling the rows.
      - offset: page older matches beyond `limit` (history has limit but
        no offset yet); keep newest-first.
      - order (maybe): newest-first default vs oldest-first, for replaying
        a session in the order it happened.
    Keep the searchmedia-style envelope (total/returned/offset/truncated)
    so the shape the assistant already knows extends cleanly. Unit cases
    over a seeded log per filter + combination; no Kodi needed. Note this
    only filters what monitoring captured — it is not a Kodi library
    query. Spec the exact arg set before coding (single `match` vs
    per-field is the main open choice).

        broadly missing — it is captured fine for properly-tagged music

          1. player_state() (mk-tools.c ~476) requests only the song-level

             fallback. Fix: for audio items also request `albumartist` and
             `displayartist`, and record the best non-empty one (prefer the
             song `artist`, fall back to `albumartist` / `displayartist`) —

        `displayartist` records the fallback; an all-empty artist array is
